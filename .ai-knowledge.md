# 🏛️ AI Knowledge & Architecture Rules (Enterprise-Grade)

Tu agis en tant que Tech Lead / Architecte Senior Zéro-Defect. Ce document est la source absolue de vérité pour ce repository. Toute modification ou réflexion doit s'y conformer **sans aucune exception**.

## 1. ARCHITECTURE DES FLUX & ROUTAGE DES TOKENS
- **API Gateway** : Point d'entrée unique. Validation des Access/Refresh Tokens clients.
- **Intercepteur gRPC** : Traduction obligatoire du token client en un `INTERNAL_TOKEN` sécurisé avant appel vers un microservice (MS).
- **Async Processing (HTTP 206)** : Renvoi immédiat d'un code `206 Partial Content` si un traitement est asynchrone. Feedback client exclusif via WebSockets (WS).


## 2. CYCLE DE VIE DES PAQUETS SÉPARÉS & CI/CD STRICTE
- **Proto-Registry** : Contrats `.proto`. Merge sur 'main' (après 'buf lint') -> màj npm via PR (version temporaire).
- **NPM Packages Internes** : 
  - TOUTE modification requiert un push Git et d'attendre la CI pour avoir une version.
  - **INTERDICTION** formelle de faire un `yarn up` avant le passage complet de la CI.
- **Versioning (Changesets)** : 
  - Obligation d'utiliser `yarn changeset add` puis `yarn changeset version`.
  - **INTERDICTION ABSOLUE** de bump deux fois le même package sur une même branche (ex: 3.1 -> 3.3 interdit).

## 3. OUTBOX PATTERN & DESIGN EVENT-DRIVEN
- **Job** = 1:1, **Event** = 1:N. Toujours utiliser `@volontariapp/messaging` pour valider les payloads.
- **Flux Absolu** :
  1. MS -> Push `jobs_outbox` (Pending).
  2. Outbox Lib -> Pull job (Processing) -> Push Redis (Done).
  3. Worker -> Pop Redis -> `job_audit` (working) -> Traitement -> `job_audit` (done).
  4. Trigger SQL -> Push `event_outbox` (Pending).
  5. Outbox Lib -> Pull event (Processing) -> Push Redis Stream (Done).
  6. Post-Processor -> Écoute Redis Stream -> Nettoie (DELETE) `jobs_outbox`.

## 4. NORMES DE CODE & TYPAGE SÉCURISÉ (ZÉRO TOLÉRANCE)
- **TypeScript Strict** : Interdiction totale de `any`, `unknown as Type` ou casts forcés.
- **Value Objects** : Interdiction d'utiliser `string`/`number` pour les données sensibles ou structurées (ID, Emails, Prix, Statuts). Utilisation stricte de classes/types Value Objects.
- **Erreurs Applicatives** : Pas de `throw new Error()`. Typage explicite des erreurs (ex: classes métier dédiées `DomainError` ou pattern `Result<SuccessData, DomainError>`).
- **Principes** : SOLID, Clean Architecture. Naming en `kebab-case`.

## 5. STRATÉGIE DE TEST NIVEAU ENTREPRISE
- **Blueprints de Fichiers** : Chaque service métier (ex: `user.service.ts`) **DOIT** être accompagné de :
  1. `user.service.spec.ts` : Assertions, describe, it uniquement.
  2. `user.service.mock.ts` : Stubs gRPC, mocks de dépendances, Spies pré-configurés.
  3. `user.factory.ts` : Utilisation de Faker ou de fixtures dures (ex: `buildUserDto()`).
- **Sécurité** : Spies obligatoires pour traquer les effets de bord et bloquer les "unbounded tests".
